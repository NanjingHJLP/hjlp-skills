# 深度研究工作流

本文件定义深度研究技能的完整执行流程。目标：在保持搜索覆盖面、来源可靠性和结果可追溯性的前提下，产出一份结构清晰、证据充分的研究报告。

## 流程总览

```text
┌─────────────────────────────────────────────────────────┐
│  Phase 1: 问题理解与查询规划                              │
│  ↓                                                       │
│  Phase 2: 广度搜索  ←──── 信息不足时补充搜索              │
│  ↓                                                       │
│  Phase 3: 深度阅读                                       │
│  ↓                                                       │
│  Phase 4: 交叉验证与整合  ←── 发现新线索时回到 Phase 2/3  │
│  ↓                                                       │
│  Phase 5: 报告生成                                       │
│  ↓                                                       │
│  Phase 6: 质量检查与交付                                  │
└─────────────────────────────────────────────────────────┘
```

---

## Phase 1：问题理解与查询规划

### 目标
将用户的研究需求转化为可执行的搜索计划。

### 执行步骤

**1.1 识别研究目标**

判断用户想要的研究类型：

| 类型 | 特征 | 示例 |
|------|------|------|
| 概念理解 | 想搞清楚某个事物是什么 | 「量子计算是什么」 |
| 对比分析 | 想比较多个选项 | 「React 和 Vue 哪个好」 |
| 趋势研判 | 想了解发展方向 | 「AI 行业未来趋势」 |
| 技术调研 | 想评估某个技术方案 | 「GraphQL 适合我们项目吗」 |
| 市场调研 | 想了解市场格局 | 「国内 CRM 市场分析」 |
| 决策支持 | 想做出选择 | 「该选哪个云服务商」 |

**1.2 确定研究范围**

明确以下边界：
- **时间范围**：需要多新的信息？（最近一年 / 历史全貌 / 特定时间段）
- **地域范围**：全球视角还是特定地区？
- **深度要求**：概览级别还是专家级别？
- **语言偏好**：中文来源为主还是中英文兼顾？

**1.3 拆解子查询**

将问题拆为 2-4 个互补的子查询，覆盖不同角度：

| 维度 | 说明 | 查询构造建议 |
|------|------|-------------|
| 基础概念 | 定义、原理、背景 | `[主题] 是什么 定义 原理` |
| 最新动态 | 近期数据、新闻、变化 | `[主题] 2026 最新 动态/数据` |
| 对比评价 | 优缺点、竞品、争议 | `[主题] vs [对手] 对比 优缺点` |
| 实践应用 | 案例、教程、经验 | `[主题] 应用场景 案例 实践` |

**示例**：用户问「了解一下 Rust 语言」

→ 子查询：
1. `Rust 编程语言 特点 优势 2026`
2. `Rust vs Go vs C++ 性能 对比`
3. `Rust 实际应用场景 企业案例`
4. `Rust 学习路线 入门难度`

**1.4 模糊问题处理**

如果用户问题过于宽泛或模糊，先澄清：

- 「你更关注哪个方面？比如技术原理、市场前景、还是实际应用？」
- 「需要多深入？概览性了解还是深度技术分析？」
- 「有特定的时间范围或地域限制吗？」

澄清后再进入 Phase 2。

### 退出条件
- [x] 研究目标类型已识别
- [x] 研究范围已确定（时间、地域、深度）
- [x] 2-4 个子查询已拆解，覆盖至少 2 个不同维度
- [x] 模糊问题已澄清（如适用）

---

## Phase 2：广度搜索

### 目标
对每个子查询分别搜索，建立信息全景。

### 执行步骤

**2.1 逐一搜索**

对 Phase 1 拆解的每个子查询，分别执行搜索：
- 每个子查询收集 5-8 条结果
- 使用搜索工具的深度阅读模式（如可用）

**2.2 记录搜索笔记**

对每条有价值的结果，记录：

```text
来源：[标题]
URL：[链接]
发布时间：[日期]
核心信息：[1-2 句关键内容]
可信度评估：高/中/低（参考 source-evaluation.md）
```

**2.3 覆盖度检查**

搜索完成后检查：
- 是否每个子查询都有独立结果？
- 来源类型是否多样？（不全来自同一个网站/平台）
- 是否有明显的信息缺口？

**2.4 补充搜索（如需要）**

以下情况需要补充搜索：
- 某个子查询结果不足 3 条
- 所有结果来自同类来源（如全是博客、全是新闻）
- 发现了新的重要维度需要覆盖
- 搜索结果中出现了意料之外的关键术语或概念

### 退出条件
- [x] 所有子查询均已独立搜索
- [x] 总计获得 15+ 条有效结果
- [x] 来源类型多样（至少覆盖 2 种来源类型）
- [x] 无明显信息缺口

---

## Phase 3：深度阅读

### 目标
从搜索结果中筛选高价值页面，提取深度信息。

### 执行步骤

**3.1 筛选高价值页面**

从 Phase 2 的结果中，按以下优先级筛选 5-8 个页面深入阅读：

| 优先级 | 来源类型 | 原因 |
|--------|----------|------|
| P0 | 官方文档、政府/学术机构 | 权威性最高 |
| P1 | 权威媒体、行业研究报告 | 数据可靠、分析专业 |
| P2 | 高质量技术博客、深度分析文章 | 有独到见解和实践经验 |
| P3 | 论坛/社区讨论 | 补充真实用户视角 |

**3.2 提取关键信息**

阅读每个页面时，重点提取：
- **明确结论**：作者的核心观点是什么？
- **关键数据**：有哪些具体数字、统计、基准测试？
- **时间信息**：数据的时间点、适用时间范围
- **适用条件**：结论在什么条件下成立？有什么限制？
- **反面证据**：有没有与主流观点不同的发现？
- **案例与实例**：有没有具体的应用案例？

**3.3 标记待验证信息**

对以下内容打上「待验证」标记：
- 只在单一来源中出现的重要数据
- 看起来过于极端或反常的结论
- 缺少具体来源的二手引用
- 利益相关方发布的评价性内容

### 退出条件
- [x] 已深读 5+ 个高价值页面
- [x] 提取了关键结论、数据、案例
- [x] 待验证信息已标记

---

## Phase 4：交叉验证与整合

### 目标
验证信息可靠性，整合为结构化笔记。

### 执行步骤

**4.1 交叉验证**

对 Phase 3 中标记为「待验证」的信息，以及所有核心事实：
- 在至少 2 个不同来源中寻找验证
- 如果找到一致证据 → 标记为「已确认」
- 如果发现矛盾 → 标记为「有争议」，记录各方观点
- 如果无法验证 → 标记为「未验证」，在报告中注明

**4.2 信息分类**

将所有收集到的信息分为三类：

| 类别 | 定义 | 报告中的处理 |
|------|------|-------------|
| 已确认事实 | 2+ 个独立来源验证 | 直接陈述，附来源 |
| 有争议观点 | 来源间存在分歧 | 呈现各方观点，标注分歧 |
| 合理推断 | 基于材料的逻辑推导 | 明确标注为推断，说明依据 |

**4.3 形成结构化笔记**

整理为以下结构：

```text
## 主题主线
[一句话概括核心发现]

## 维度 1：[名称]
- 已确认：...
- 有争议：...
- 关键数据：...（来源：...）

## 维度 2：[名称]
- 已确认：...
- 有争议：...
- 关键数据：...（来源：...）

## 待补充
- [如有信息缺口，记录在此]
```

**4.4 决策分支**

根据整合结果判断：
- 信息充分 → 进入 Phase 5
- 发现重要新线索 → 回到 Phase 2 补充搜索
- 某个维度深度不足 → 回到 Phase 3 补充阅读

### 退出条件
- [x] 核心事实已交叉验证
- [x] 信息已分类（已确认 / 有争议 / 合理推断）
- [x] 结构化笔记完成
- [x] 无重大信息缺口

---

## Phase 5：报告生成

### 目标
将结构化笔记转化为正式研究报告。

### 执行步骤

**5.1 选择报告模板**

根据研究类型选择合适的模板（详见 [references/report-template.md](references/report-template.md)）：

| 研究类型 | 推荐模板 |
|----------|----------|
| 概念理解 / 技术调研 | 标准研究报告 |
| 对比分析 / 决策支持 | 对比分析报告 |
| 趋势研判 / 市场调研 | 趋势分析报告 |

**5.2 撰写报告**

写作要求：
- **概述先行**：先给结论，再展开细节
- **论据充分**：每个关键论点附带来源
- **数据精确**：标注来源和时间
- **分歧透明**：有争议内容如实呈现各方观点
- **语言匹配**：报告语言跟随用户语言

**5.3 保存报告**

- 文件名格式：`[主题关键词]-研究报告-[日期].md`
- 保存到用户桌面（或用户指定位置）

### 退出条件
- [x] 报告结构完整（概述、分析、发现、来源）
- [x] 所有关键论点附带来源
- [x] 报告已保存到文件

---

## Phase 6：质量检查与交付

### 目标
确保报告质量达标，交付给用户。

### 执行步骤

**6.1 执行质量检查清单**（见下方）

**6.2 交付**
- 告知用户文件保存路径
- 简要概述核心发现（3-5 句话）
- 询问是否需要补充某个方向或调整深度

### 退出条件
- [x] 质量检查清单全部通过
- [x] 用户已收到报告路径和核心摘要

---

## 质量检查清单

在交付报告前，逐项检查：

### 覆盖度
- [ ] 研究覆盖了 2+ 个维度
- [ ] 执行了多轮搜索（非单轮）
- [ ] 深读了 5+ 个关键来源

### 可靠性
- [ ] 核心事实至少 2 源交叉验证
- [ ] 有争议信息已标注分歧
- [ ] 未验证信息已注明

### 可追溯性
- [ ] 每个关键论点附带来源链接
- [ ] 数据标注了来源和时间
- [ ] 报告末尾有完整参考来源列表

### 报告质量
- [ ] 概述部分能独立传达核心发现
- [ ] 结论与推测明确区分
- [ ] 不是搜索结果的简单堆砌
- [ ] 语言与用户语言一致

### 交付完整性
- [ ] 报告已保存到文件
- [ ] 已告知用户文件路径
- [ ] 已提供核心发现摘要

---

## 异常处理

| 异常情况 | 处理方式 |
|----------|----------|
| 搜索工具不可用 | 告知用户当前环境限制，建议切换到有搜索能力的环境 |
| 搜索结果极少（< 5 条） | 尝试换关键词、放宽时间范围、使用英文搜索 |
| 页面无法访问 | 跳过该页面，从其他来源补充 |
| 所有来源高度一致 | 警惕信息茧房，主动搜索反面观点 |
| 所有来源高度矛盾 | 如实呈现分歧，不强行给出单一结论 |
| 用户中途调整方向 | 回到 Phase 1 重新规划，已有材料可复用 |
| 研究范围过大 | 与用户协商缩小范围，或拆分为多次研究 |
